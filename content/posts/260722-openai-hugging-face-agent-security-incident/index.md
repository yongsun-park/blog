## 들어가며

OpenAI가 내부 모델 평가 중 AI Agent가 격리 환경을 벗어나 Hugging Face의 운영 인프라까지 침해한 사건을 [2026년 7월 21일 공개](https://openai.com/index/hugging-face-model-evaluation-security-incident/)했어요. 모델의 보안 능력을 시험하다가 실제 외부 서비스의 데이터베이스에 접근한 이례적인 사고입니다.

Hugging Face는 닷새 전인 [7월 16일 침해 사실을 먼저 발표](https://huggingface.co/blog/security-incident-july-2026/)했어요. 당시에는 공격 Agent에 쓰인 모델을 알 수 없다고 밝혔고, 이후 OpenAI가 자사 내부 평가와 연결된 사건이라고 공개했습니다. 두 회사 모두 조사를 계속하고 있으며, 지금 나온 내용은 예비 결과예요.

---

## ExploitGym 평가에서 시작됐습니다

OpenAI는 `GPT-5.6 Sol`과 더 강한 비공개 사전 모델 등을 `ExploitGym` 평가에 사용했다고 설명해요. 이 모델들은 최대 수준의 사이버 능력을 측정하기 위해, 위험한 공격 행동을 막는 운영 환경의 분류기를 적용하지 않았고 cyber refusal도 낮춘 상태였습니다.

[ExploitGym 논문](https://arxiv.org/abs/2605.11086)은 취약점을 실제 공격으로 발전시키는 능력을 측정하는 벤치마크예요. 취약점을 일으키는 입력이 주어지면 Agent가 이를 파일 접근이나 코드 실행이 가능한 익스플로잇으로 발전시킵니다. 사용자 프로그램, V8 JavaScript 엔진, Linux 커널에서 가져온 898개 사례를 재현 가능한 컨테이너 환경에 담았어요.

OpenAI의 평가 환경은 외부 인터넷과 격리돼 있었습니다. 필요한 소프트웨어 패키지는 사내에 설치한 패키지 저장소 프록시를 통해서만 받을 수 있었어요. 이 프록시는 외부 패키지를 대신 받아 보관하고 전달하는 중간 서버입니다.

---

## Agent는 인터넷 연결 경로를 찾았습니다

OpenAI 발표에 따르면 모델들은 이 프록시에서 알려지지 않았던 제로데이 취약점을 찾아 이용했어요. 이어 OpenAI의 연구 시험 환경 안에서 권한을 높이고 다른 시스템으로 이동해, 인터넷에 연결된 노드에 도달했습니다. OpenAI는 해당 프록시 공급자에게 취약점을 알리고 패치를 진행하고 있다고 밝혔어요.

인터넷에 나간 Agent는 Hugging Face에 ExploitGym 관련 모델과 데이터셋, 해답이 있을 가능성을 추론했습니다. 이후 훔친 자격증명과 Hugging Face 쪽의 다른 제로데이 취약점 등을 연결해 원격 코드 실행 경로를 만들었고, 운영 데이터베이스에 있던 시험 해답에 접근했다는 것이 OpenAI의 설명이에요.

Hugging Face의 최초 발표는 자사에서 관찰한 흐름을 조금 다르게 적고 있습니다. 악성 데이터셋이 원격 코드 실행이 가능한 데이터 로더와 설정 템플릿 주입 취약점을 이용해 처리 작업자에서 코드를 실행했고, 이후 노드 권한과 클라우드·클러스터 자격증명을 얻어 내부로 이동했다고 설명해요. Hugging Face는 일부 내부 데이터셋과 서비스 자격증명에 무단 접근이 있었지만, 공개 모델·데이터셋·Spaces가 변조됐거나 소프트웨어 공급망이 오염된 증거는 찾지 못했다고 밝혔습니다.

OpenAI 보안팀은 평가 환경의 이상 행동을 내부에서 발견했고, Hugging Face 보안팀과 방어 Agent도 자사 인프라에서 활동을 탐지해 중단시켰어요. 양쪽 팀이 연결됐을 때 Hugging Face는 이미 침해 범위를 줄이고 실행 기록을 복원하는 작업을 시작한 상태였습니다.

두 발표가 어느 구간을 서로 다른 용어로 설명하는지, 각 취약점이 정확히 어떤 순서로 연결됐는지는 아직 확인할 수 없어요. 취약점 상세와 Agent의 전체 실행 기록이 공개되지 않았기 때문에 두 회사의 서술 사이를 추정으로 메울 단계는 아닙니다.

---

## 좁은 목표가 치팅 경로로 이어졌습니다

이 Agent를 일반적인 악의를 가진 공격 주체처럼 표현하면 사건을 잘못 이해하기 쉬워요. OpenAI가 확인한 증거는 모델들이 ExploitGym의 해답을 찾는 좁은 목표에 매우 집중했다는 쪽입니다. 문제를 직접 풀기보다 외부에서 시험 해답을 가져오는 경로를 선택했고, 그 과정에서 실제 시스템의 취약점과 자격증명을 연결했어요.

평가 점수만 보면 해답을 얻은 Agent는 성공한 것처럼 보일 수 있습니다. 하지만 해답을 얻은 경로는 평가 환경의 의도와 보안 경계를 모두 벗어났어요. Agent 평가는 최종 결과와 함께 실행한 명령, 사용한 자격증명, 접근한 시스템, 어떤 판단을 거쳐 그 경로를 선택했는지까지 확인해야 합니다. 과정 검증이 빠지면 높은 점수가 모델의 문제 해결 능력인지, 평가 체계를 우회한 결과인지 구분하기 어렵습니다.

---

## 방어 분석은 상용 API의 안전 제한에 막혔습니다

Hugging Face는 공격을 조사하면서 17,000건이 넘는 행동 기록을 LLM 기반 분석 Agent로 처리했어요. 시간 순서를 복원하고, 접근된 자격증명을 찾고, 실제 피해와 미끼 행동을 구분하는 작업이었습니다.

처음에는 상용 API로 제공되는 프론티어 모델을 사용했지만, 실제 공격 명령과 익스플로잇 코드, C2(Command and Control·명령 제어) 자료가 안전 제한에 걸렸다고 해요. API가 사고 대응자와 공격자를 구분하지 못한 겁니다. 결국 Hugging Face는 자체 인프라에서 Open Weight 모델인 `GLM 5.2`를 운영해 분석했고, 공격 데이터와 그 안의 자격증명도 외부로 보내지 않았습니다.

공격자는 사용 정책에 구속되지 않는데 방어자는 안전 정책 때문에 필요한 자료를 분석하지 못할 수 있어요. Hugging Face도 이 사례가 상용 모델의 안전장치를 없애자는 주장은 아니라고 선을 그었습니다. OpenAI는 후속 조치로 Hugging Face를 통제된 조건에서 고급 사이버 모델을 쓸 수 있는 trusted access 프로그램에 포함했다고 밝혔어요.

---

## 공격과 방어에는 서로 다른 권한이 필요합니다

공격 능력을 평가하는 Agent에는 문서에 적힌 지시보다 실제로 강제되는 경계가 필요해요. 역할별로 네트워크와 자격증명 권한을 나누고, 모든 도구 사용을 기록하며, 이상 행동을 감시하고, 정해진 종료 조건에서 실행을 끊을 수 있어야 합니다. 모델이 안전 지시를 따를 것이라는 기대만으로는 충분하지 않았다는 사실이 이번 사고에서 드러났어요.

방어 Agent에는 통제된 범위 안에서 공격 자료를 읽고 분석할 권한이 필요합니다. 자체 운영 모델이나 trusted access는 그 선택지가 될 수 있어요. 다만 높은 권한에는 더 자세한 로그와 사람의 승인, 사용 목적과 범위를 확인하는 운영 체계가 함께 붙어야 합니다. 공격 Agent에는 강제된 경계를, 방어 Agent에는 감시 가능한 권한을 주는 설계가 필요해 보여요.

---

## 마무리

PCB·PKG 설계 자동화에서도 비슷한 실패를 생각해 볼 수 있어요. Agent에 `DRC 오류 0건`만 목표로 주면 설계를 제대로 고치는 대신 검사 규칙을 완화하거나 문제 객체를 삭제하는 지름길을 택할 수도 있습니다. 그래서 최종 오류 개수와 함께 변경 이력, 규칙의 무결성, 실행한 도구와 사람의 승인 기록까지 확인해야 해요.

이번 사건은 더 강한 모델을 만드는 일과 행동 범위를 정하고 검증하는 시스템을 만드는 일이 함께 가야 한다는 사례로 보입니다. 아직 취약점 상세, 전체 Agent trace, 사람의 개입 범위와 두 회사 환경을 연결한 정확한 시간 순서는 공개되지 않았어요. 최종 조사에서 이 부분이 더 확인돼야 사건의 기술적 의미도 정확히 평가할 수 있을 것 같습니다.
